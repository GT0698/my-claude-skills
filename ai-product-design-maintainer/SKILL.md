# AI 产品设计 Skill 维护

专门负责读取课件和课堂笔记，审计、更新并验证 `ai-product-design`。不要在日常产品方案讨论中加载本流程。

## 来源原则

- 课件和课堂笔记都是正式来源：课件是完整性锚点，笔记是经过课堂理解和重组的主要蒸馏对象。
- 不能只根据其中一类更新。
- 课程内容、用户补充和维护者推断必须区分；非课程补充应明确标注。
- 简历、求职等用户明确排除的课件不得并入产品设计方法论。

## 更新单篇材料

1. 读取新课件的全部文字、图片和图表。
2. 查找并读取对应课堂笔记；没有笔记时明确记录“仅课件来源”。
3. 读取目标 Skill 的 `SKILL.md`、相关 reference 和 `SOURCES.md`。
4. 做差异分析：新增概念、深化内容、冲突、重复、过时结论和来源缺口。
5. 将稳定的方法、判断标准和流程并入最相关的 reference；不要把整篇课程摘要塞进 Skill。
6. 更新 `SOURCES.md` 登记和课程总数等易陈旧信息。
7. 运行格式校验和回归测试，汇报改动与仍待确认的问题。

## 全量复盘

读取 `ai-product-design/references/维护与复盘.md` 并执行完整流程：获取课件和笔记实际清单，与来源登记做双向差集，核对已登记内容是否进入 reference，并检查触发边界、引用路径、重复、过时工具和矛盾。

## 结构约束

- `ai-product-design/SKILL.md` 只保留方案讨论的核心流程与 reference 路由。
- 详细课程知识放在对应 reference，按需加载。
- 维护流程和来源登记不回填到日常 Skill 主体。
- 新增模块前先判断能否归入已有主题，避免 reference 无限膨胀。

## 验证与交付

- 运行 Skill 基础格式校验并检查所有本地链接存在。
- 至少测试三类请求：应触发产品设计、不应触发产品设计、产品设计后应交接 PRD。
- 对比改动前后的回答，检查是否减少无关框架、追问和误触发。
- 说明读取了哪些课件和笔记、修改了哪些文件、哪些内容被排除、校验结果和未解决项。
- 没有证据的推断不得写成课程结论。
